# Memory-First Rule

AT SESSION START:
Read your shared brain file (path set in CLAUDE.md) before anything
else — project states, last session, and feedback log. The session start
checklist lives there.

Then read the active project's CLAUDE.md for project-specific context.

BEFORE ASKING THE HUMAN A QUESTION:
Check if the answer exists in the playbook or project memory. If it does,
use it. Don't ask what you already know.

BEFORE STARTING WORK:
Confirm a spec exists (micro or full). If none exists, write one first.
Exception: exploration sessions (research, brainstorming, debugging) need
a trigger question instead: "What are we trying to learn or decide?"

ON CORRECTION (the human says "no, not that" or "don't do X"):
Write a feedback memory IMMEDIATELY — before your next response.
Add it to the Feedback Log in your shared brain. Not later. Not "at the end." Now.

ON CONFIRMATION (the human says "yes exactly" or accepts non-obvious approach):
Write that down too. Learn from success, not just failure.

ON PROJECT STATE CHANGE:
Update the playbook's Active Projects table before moving on.

CROSS-PROJECT:
When a pattern proves itself in one project, check if it belongs in the
shared brain's permanent rules. Patterns that survive 3+ projects are
candidates for a new rule file.

MID-SESSION CONTEXT HYGIENE:
Watch for these pressure signals:
- Same problem attempted 3+ times without progress
- 10+ consecutive tool calls without responding to the human
- Reading more than 5 large files (500+ lines each) in one session
- A growing sense of "let me try one more thing"

When you notice any signal: STOP.
1. Summarize what's been done and what's stuck — out loud, to the human
2. Propose: clear direction forward, OR capture state and start fresh
3. Never spiral silently deeper into a broken approach

If a session is getting long after a major chunk completes:
"Good stopping point — want to capture state before we continue?"
One sentence. Not an interruption.
